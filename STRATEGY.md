# Black Friday Rolling Migration Strategy

## Black Friday Rolling Migration

**Стратегия экономии на серверах с возможностью масштабирования**

Инфраструктура для pet-проектов: **Docker Swarm + Dokploy + Ansible + Tailscale**

---

# 1. Суть стратегии

Идея проста: вместо долгосрочной аренды дорогого сервера — покупать дешёвые VPS на распродажах (Black Friday, Cyber Monday), временно держать два сервера в одном кластере, мигрировать данные и приложения, а потом не продлевать старый дорогой сервер.

Это не хак и не костыль — это осознанная стратегия **rolling migration**, адаптированная под реальность pet-проектов с минимальным бюджетом.

### Цель

Сэкономить на серверах без потери надёжности и масштабируемости.

### Целевая аудитория

Разработчик с **1–3 pet-проектами** с небольшой нагрузкой.

### Ключевой принцип

**Overlap period** — оба сервера живут вместе ~6 месяцев.

### Инструменты

* Docker Swarm
* Dokploy
* Ansible
* Tailscale
* GitLab

---

# 2. Стек технологий и роли компонентов

## Docker Swarm

Оркестратор контейнеров. Позволяет объединить несколько серверов в один кластер и запускать сервисы распределённо.

Для одного сервера — минимальный overhead, но стек сразу готов к добавлению новых нод.

---

## Dokploy

UI для деплоя поверх Docker Swarm.

Устанавливается на **manager-ноде**, подключается к GitLab, тянет изменения и деплоит сервисы.

Хранит конфигурацию проектов и env-переменные в локальной PostgreSQL.

⚠️ Важно:

Dokploy **не синхронизируется между нодами** — это инструмент управления, а не сервис реального времени.

При смене сервера просто настраивается заново вручную (20–30 минут для 2–3 проектов).

---

## Ansible

Автоматизация раскатки серверов.

Обеспечивает идентичную конфигурацию на всех нодах:

* Docker
* Tailscale
* системные пакеты
* firewall

Один раз написал playbook — применяешь на любом новом VPS.

---

## Tailscale

Mesh-VPN на базе **WireGuard**.

Объединяет серверы у разных провайдеров в одну приватную сеть.

Каждая нода получает стабильный адрес:

```
100.x.x.x
```

Docker Swarm работает поверх этой сети, не зная о провайдерах.

Бесплатный план:

* до **3 пользователей**
* до **100 устройств**

Для pet-проектов этого более чем достаточно.

---

# 3. Текущая архитектура (один VPS)

## Схема: один сервер

```
Ansible
   ↓
1 VPS
   ↓
Docker Swarm (single-node manager)
   ↓
Dokploy
   ↓
GitLab
```

### Компоненты

* **VPS** — один сервер у любого провайдера
* **Swarm** — инициализирован в single-node режиме
* **Tailscale** — установлен и готов к добавлению нод
* **Dokploy** — UI для деплоя, настроен на GitLab-репозитории

Tailscale на одном сервере не даёт операционных преимуществ, но его наличие — архитектурное решение: инфраструктура заложена под масштабирование с первого дня.

---

# 4. Сценарий Black Friday — пошаговый план

## Шаг 1 — Покупка нового VPS на распродаже

Провайдеры дают скидки **50–80%** в Black Friday / Cyber Monday.

Покупаем VPS аналогичного уровня по выгодной цене.

Старый сервер продолжает работать.

---

## Шаг 2 — Раскатка нового сервера через Ansible

```bash
ansible-playbook -i inventory setup.yml --limit new_server
```

Ansible устанавливает:

* Docker
* Tailscale
* системные пакеты
* firewall

Новый сервер получает **идентичную конфигурацию** со старым.

---

## Шаг 3 — Подключение в Tailscale сеть

```bash
tailscale up
```

Новый сервер авторизуется в Tailscale и получает адрес `100.x.x.x`.

Оба сервера теперь видят друг друга в приватной сети.

---

## Шаг 4 — Добавление в Swarm кластер

Получить токен на старом сервере:

```bash
docker swarm join-token worker
```

На новом сервере:

```bash
docker swarm join --token <token> 100.x.x.x:2377
```

Promote до manager:

```bash
docker node promote <новый_сервер>
```

⚠️ Это критически важно.

Если новый сервер — только **worker**, при отключении старого кластер потеряет управление.

---

## Шаг 5 — Миграция базы данных

Это **самый важный шаг**.

БД живёт на старом сервере — нужно перенести её на новый.

### PostgreSQL

На старом сервере:

```bash
pg_dump -U postgres mydb > mydb.sql
scp mydb.sql user@100.новый.сервер:/tmp/
```

На новом сервере:

```bash
psql -U postgres mydb < mydb.sql
```

### Момент переключения

Для pet-проекта:

* делаем в момент минимальной нагрузки.

Для продакшена:

* включаем **maintenance mode**
* делаем финальный dump
* переключаем.

Время простоя — **5–10 минут**.

---

## Шаг 6 — Настройка Dokploy на новом сервере

Устанавливаем Dokploy.

Вручную:

* создаём проекты
* прописываем env-переменные

Обычно это занимает **20–30 минут для 2–3 проектов**.

💡 Совет:

Держи env-переменные в **GitLab CI/CD variables** — тогда настройка нового Dokploy займёт ещё меньше времени.

---

## Шаг 7 — Переключение DNS и проверка

Меняем **A-записи DNS** на IP нового сервера.

Проверяем:

* сервисы
* базы
* деплой

Рекомендуется наблюдать **1–2 недели**.

---

## Шаг 8 — Отключение старого сервера

Просто **не продлеваем сервер**.

Он отключается.

Кластер продолжает работать на новом сервере.

**Цель достигнута.**

---

# 5. Overlap Period — 6 месяцев вместе

Полгода — огромный запас времени.

### Месяц 1

Раскатка нового сервера и подключение в кластер.

### Месяц 1–2

Миграция БД и настройка Dokploy.

### Месяц 2–3

Тестирование всех сервисов.

### Месяц 3–5

Период наблюдения — оба сервера работают.

### Месяц 6

Старый сервер не продлеваем.

**Profit.**

---

# 6. Масштабирование при росте нагрузки

Если нагрузка выросла:

1. Покупаем второй VPS
2. Ansible настраивает сервер
3. Подключаем к Tailscale и Swarm
4. Теперь два сервера работают постоянно
5. Swarm распределяет контейнеры автоматически

⚠️ Важно при multi-node:

Volumes **не шарятся автоматически**.

Нужно:

* использовать S3 / managed DB
  или
* фиксировать сервис на конкретной ноде.

---

# 7. Ключевые принципы стратегии

## Принцип 1 — Overlap First

Никогда не отключай старый сервер, пока новый полностью не работает.

**6 месяцев — буфер безопасности.**

---

## Принцип 2 — Manager Promotion

Новый сервер **обязательно promote до manager**.

Иначе при отключении старого сервера кластер потеряет управление.

---

## Принцип 3 — Data First

Данные мигрируют **первыми**.

До:

* переключения DNS
* отключения старого сервера

БД — самое ценное.

Всё остальное можно восстановить из GitLab.

---

## Принцип 4 — Ansible Idempotency

Playbook должен давать **одинаковый результат на любом сервере**.

Это гарантия идентичности инфраструктуры.

---

## Принцип 5 — Простота > автоматизация

Для **2–3 проектов** сложная автоматизация не нужна.

Настроить вручную за **30 минут** лучше, чем писать хрупкие скрипты.

---

# 8. Чеклист миграции

1. Купить новый VPS
2. Запустить Ansible playbook
3. Авторизовать сервер в Tailscale
4. Добавить в Swarm
5. Promote до manager
6. Сделать `pg_dump`
7. Восстановить БД
8. Установить Dokploy
9. Прописать env-переменные
10. Переключить DNS
11. Наблюдать 1–2 недели
12. Не продлевать старый сервер ✓
